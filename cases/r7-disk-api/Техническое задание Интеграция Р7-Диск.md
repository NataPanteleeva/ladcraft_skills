# Техническое задание

**Разработка навыка Ladcraft «r7-disk-api» — интеграция с Р7-Диск (KS 2024 API)**

| | |
|---|---|
| **Версия документа** | 1.0 |
| **Дата** | 31.05.2026 |
| **Платформа** | Ladcraft (AI-агент, MCP-навыки) |
| **Внешняя система** | Р7-Диск, REST API КС 2024 |

| **Нормативная база API** | `instruction-for-ks-2024-api-disk.pdf`, Swagger `{BASE_URL}/docs` |

---

## 1. Общие сведения

### 1.1. Назначение разработки

Требуется разработать и сопровождать **навык Ladcraft** для платформы AI-агента, обеспечивающий интеграцию с **сервером Р7-Диск** через официальный REST API КС 2024.

Навык должен представлять собой **изолированный пакет** (`SKILL.md`, скрипты tools, справочник API), пригодный для установки на различных стендах Р7-Диск с настройкой через переменные окружения навыка без изменения исходного кода.

**Основное назначение решения:**

- авторизация агента в Р7-Диске и кэширование токена;
- просмотр структуры папок и списка документов;
- создание, загрузка, перемещение, копирование, удаление и восстановление папок и файлов в пределах API;
- создание текстовых и простых DOCX-документов из чата агента **без** pip, python-docx и локального Python;
- корректная работа с кириллическими именами файлов и содержимым UTF-8;
- явные ответы агента на ограничения API (в т.ч. невозможность переименования папок).

### 1.2. Границы решения

Навык `r7-disk-api` покрывает только интеграцию с API Р7-Диска и не включает внешние бизнес-системы, локальные формы приложений и хранение прикладных связей между объектами сторонних систем.

В рамках настоящего ТЗ рассматриваются:

- API-авторизация и выполнение операций с папками/файлами;
- инструкции агенту для корректного диалога с пользователем;
- технические ограничения API и стандартизированные ответы tools.

---

## 2. Общая концепция решения

Навык поставляется как **набор MCP-tools** и инструкций для агента, содержащий:

- подсистему авторизации (`r7_disk_login`);
- инструменты просмотра (`r7_disk_list_directory`, `r7_disk_browse`);
- инструмент управления папками (`r7_disk_folder`);
- инструмент управления документами (`r7_disk_document`);
- справочник эндпоинтов (`reference.md`);
- сценарий работы агента (pipeline в `SKILL.md`).

Решение **не привязано** к конкретному хосту: базовый URL, учётные данные и опционально ID корневой папки задаются при установке навыка. Список `resources.network.hosts` в meta-файлах — **белый список** доменов для `fetch`, а не адрес сервера по умолчанию.

**Принцип разделения ответственности:**

| Слой | Ответственность |
|------|-----------------|
| **Код навыка (JS)** | HTTP-запросы к API, кодирование файлов, сборка DOCX, гарантированные ответы tool (`impossible`, `needs_credential_choice`) |
| **SKILL.md / mcp_spec** | Поведение агента: вопросы пользователю, запрет pip, формат вывода, pipeline |
| **Переменные environment.user** | Сохранённые учётные данные и URL стенда |

## 3. Основные сценарии использования

### 3.1. Работа пользователя через AI-агента

**Например:**

Пользователь в чате Ladcraft пишет: «Покажи содержимое папки Ладкрафт» или «Создай файл Отчёт.docx с текстом …».

Агент:

1. Уточняет источник учётных данных (если в навыке заданы `R7_DISK_*`) — см. п. 4.2.
2. Вызывает `r7_disk_login` с `credential_source`.
3. Вызывает `r7_disk_list_directory` (корень «Мои документы» определяется автоматически после login).
4. Формирует для пользователя таблицу подпапок и документов (Id, Name, Size, дата).
5. При создании файла — только `r7_disk_document` (`create` / `upload` с `content_text` или `content_base64`).

**При попытке переименовать папку:**

Пользователь: «Переименуй папку test в тест».

Агент вызывает `r7_disk_folder` с `action: rename` (или не вызывает API переименования документа для папки). Tool возвращает `ok: false`, `impossible: true`. Агент сообщает пользователю, что **действие невозможно** в API, и предлагает переименование в веб-интерфейсе Р7-Диска **только по явному запросу** пользователя.

### 3.2. Настройка и сопровождение администратором

**Например:**

Администратор устанавливает навык на стенд GPTZ:

1. Заполняет `R7_DISK_BASE_URL` = `https://cddisk.gptz.lad-soft.ru` (не URL веб-админки `admin.*`).
2. Задаёт `R7_DISK_LOGIN`, `R7_DISK_PASSWORD`.
3. Папку «Мои документы» настраивать не нужно — определяется автоматически после login.
4. Добавляет хост API в `resources.network.hosts` всех `scripts/*.meta.md`.
5. Выполняет deploy/update навыка на платформе.

Пользователь с другим стендом в том же чате может выбрать «другой диск» — агент передаёт `credential_source: custom` и параметры `base_url`, `login`, `password`.

## 4. Функциональные требования

### 4.1. Навык `r7-disk-api`

В репозитории Ladcraft должен быть размещён каталог навыка:

```
skills/r7-disk-api/
├── SKILL.md
├── reference.md
├── Техническое задание Интеграция Р7-Диск.md
└── scripts/
    ├── r7_disk_login.js / .meta.md
    ├── r7_disk_list_directory.js / .meta.md
    ├── r7_disk_browse.js / .meta.md
    ├── r7_disk_folder.js / .meta.md
    └── r7_disk_document.js / .meta.md
```

Навык должен проходить валидацию платформы (dry-run deploy) без блокирующих ошибок.

### 4.2. Авторизация и выбор источника учётных данных

**Tool:** `r7_disk_login`  
**API:** `POST {BASE_URL}/api/v2/auth/Login`

| Требование | Реализация |
|------------|------------|
| Получение `AuthToken` | Парсинг `Response.Data.Tokens.AuthToken` |
| Проверка доступа к Диску | Наличие `"Disk"` в `modules_access` |
| Кэш токена | `skillStorage`: `r7_disk_auth_token`, `r7_disk_base_url`, `r7_disk_expired_at` |
| Выбор: сохранённые переменные или другой диск | Параметр `credential_source`: `environment` \| `custom` |
| Запрет «тихого» login при заполненном environment | Без `credential_source` — ответ `needs_credential_choice: true` и `environment_preview` |
| Инструкция агенту задать вопрос | Раздел «Шаг 1 — Авторизация» в `SKILL.md` |

**Переменные `environment.user`:**

| Переменная | Обязательность | Назначение |
|------------|----------------|------------|
| `R7_DISK_BASE_URL` | да* | Базовый URL API (без `/docs`, без завершающего `/`) |
| `R7_DISK_LOGIN` | да* | Логин |
| `R7_DISK_PASSWORD` | да* | Пароль (`secret: true`) |

\* Не обязательны при каждом вызове, если пользователь выбрал `credential_source: custom` и передал параметры явно.

**Порядок разрешения `base_url`:** параметр вызова → кэш после login → `R7_DISK_BASE_URL`.

### 4.3. Работа с папками (DocumentDirectory)

**Tool:** `r7_disk_folder`

| action | API | Требование |
|--------|-----|------------|
| `create` | POST AddSubDirectory | Создание по `parent_directory_id` + `name` или цепочки `folder_path` |
| `move` | GET Move | `folder_id`, `to_directory_id` |
| `copy` | GET Copy | `rule` 0 / 1 / 2 |
| `delete` | DELETE Delete | `folder_id` |
| `restore` | POST Restore | `folder_id` / `folder_ids` |
| `conflict` | POST Conflict | Разрешение конфликта при копировании |
| `rename` | — | **Не реализуется.** Ответ: `impossible: true`, сообщение о невозможности |

**Просмотр:**

| Tool | API | Назначение |
|------|-----|------------|
| `r7_disk_list_directory` | GET DocumentDirectory/Get | Одна папка: `folders`, `documents`, `parent`, `counters` |
| `r7_disk_browse` | Рекурсивный обход Get | Дерево + `all_documents`; `max_depth` по умолчанию 5 в коде |

### 4.4. Работа с документами (Documents)

**Tool:** `r7_disk_document`

Навык должен обеспечивать (в пределах KS 2024 API):

| action | Назначение |
|--------|------------|
| `create` | Пустой файл или с `content_text` / `content_base64` |
| `upload` | Загрузка в папку (multipart, UTF-8 имена) |
| `rename` | Переименование **только файла** (`document_id`, `name`) |
| `move`, `copy` | Перемещение / копирование |
| `delete`, `restore` | Удаление / восстановление |
| `exists`, `get_id_by_name` | Поиск по имени в папке |
| `download` | Скачивание (лимит **10 МБ**) |
| `versions`, `change_version` | Версии документа |
| `convert` | Конвертация (например, в PDF) |

**Создание DOCX из чата:**

- При `name` / `file_name` с расширением `.docx` и `content_text` навык собирает минимальный DOCX (ZIP/OOXML) **в JavaScript**, без внешних библиотек и без сетевых запросов к `schemas.openxmlformats.org` в исходнике (требование whitelist Ladcraft).
- Поддерживаемая разметка в `content_text`: `**жирный**`, `*курсив*`, перенос строки — новый абзац.
- Сложное форматирование — только через загрузку готового `.docx` в `content_base64`.

**Запрещено для агента (инструкция в SKILL.md):** создавать файлы на Диске через pip, python-docx, локальный Python или shell.

### 4.5. Инструменты (tools) платформы Ladcraft

| Tool | Скрипт | Таймаут (тип.) |
|------|--------|----------------|
| `r7_disk_login` | `r7_disk_login.js` | 30 с |
| `r7_disk_list_directory` | `r7_disk_list_directory.js` | 30 с |
| `r7_disk_browse` | `r7_disk_browse.js` | 60 с |
| `r7_disk_folder` | `r7_disk_folder.js` | 60 с |
| `r7_disk_document` | `r7_disk_document.js` | 60 с |

Каждый tool описан в `mcp_spec` (секция `SKILL.md`) и в соответствующем `scripts/*.meta.md` (схемы input/output, `resources.network.hosts`).

### 4.6. Ограничения API и поведение агента

| Ограничение | Поведение навыка / агента |
|-------------|---------------------------|
| Нет переименования папок | `r7_disk_folder` → `rename` → `impossible: true` |
| Нет Bearer в Authorization | Заголовок `Authorization: <AuthToken>` |
| Веб-URL ≠ API-URL | В SKILL.md таблица: `admin.*` — нет, `cddisk.*` — да |
| Кириллица в upload | `content_text` + RFC 5987 `filename*=UTF-8''…`, тело как `Uint8Array` |
| Лимит download | 10 МБ; иначе сообщение об ошибке |
| Длинный base64 в чате | Не выводить без запроса пользователя |

## 5. Требования к подключению

Для подключения навыка администратор / разработчик должен:

1. Разместить каталог `r7-disk-api` в `skills/` проекта Ladcraft.
2. Указать в установке навыка переменные `environment.user` (см. п. 4.2).
3. Включить в `resources.network.hosts` каждого скрипта фактический хост из `R7_DISK_BASE_URL`.
4. Выполнить deploy/update навыка на целевой платформе.
5. Убедиться, что у учётной записи есть модуль `Disk` в `modules_access`.

**Типовые ошибки подключения** (должны быть описаны в SKILL.md):

| Симптом | Действие |
|---------|----------|
| `401 Unauthorized` | Проверить логин/пароль, повторить login |
| Запросы на неверный хост | Исправить `R7_DISK_BASE_URL`, redeploy |
| `Invalid network whitelist host` | Добавить хост в meta, redeploy |
| Login OK, затем ошибка whitelist | Проверить `api_base_url` в ответе; обновить навык |

---

## 6. Требования к безопасности

Должны быть предусмотрены:

- хранение пароля только в `environment.user` с флагом `secret: true`, не в коде навыка;
- не выводить в чат `auth_token`, пароль, полный JSON ответа без запроса;
- использование HTTPS для `R7_DISK_BASE_URL` на продуктивных стендах;
- работа от имени пользователя, указанного в login (права Р7-Диска);
- кэш токена в `skillStorage` в рамках сессии навыка;
- явный выбор стенда (`credential_source`) при наличии сохранённых учётных данных.

---

## 7. Требования к расширяемости

Система должна предусматривать:

- добавление новых хостов в `network.hosts` без изменения логики tools;
- расширение `r7_disk_document` новыми action при появлении методов в API КС 2024;
- дополнение `reference.md` при обновлении PDF/Swagger;
- опциональные параметры tools (`auth_token`, `base_url`) для отладки без смены environment.

## 9. Ожидаемый результат

В результате разработки и внедрения навыка `r7-disk-api` должна быть обеспечена возможность:

- выполнять из чата Ladcraft типовые операции с Р7-Диском без ручных REST-вызовов;
- централизованно настраивать подключение к стенду через переменные навыка;
- переключаться на другой стенд по запросу пользователя (`credential_source: custom`);
- создавать текстовые и простые DOCX-файлы с кириллицей напрямую на Диске;
- получать однозначные ответы об ограничениях API (переименование папок, отсутствие модуля Disk);
- минимизировать ошибки агента (pip/python-docx, путаница admin URL и API URL).

## 10. Критерии приёмки

| № | Критерий | Метод проверки |
|---|----------|----------------|
| 1 | Dry-run deploy навыка без блокирующих ошибок | `POST /api/remote/deploy-local/dry-run` |
| 2 | Login с `credential_source: environment` на тестовом стенде | Вызов tool, `ok: true`, `"Disk"` в `modules_access` |
| 3 | Без `credential_source` при заполненном env — `needs_credential_choice` | Вызов tool, проверка поля ответа |
| 4 | List папки с кириллическим именем | `r7_disk_list_directory`, корректные `Name` в ответе |
| 5 | Upload/create файла `.txt` и `.docx` с кириллицей | Открытие в веб-интерфейсе Р7-Диска |
| 6 | DOCX с `**жирный**` открывается в Word | Визуальная проверка |
| 7 | `r7_disk_folder` action `rename` → `impossible: true` | Вызов tool, проверка текста ошибки |
| 8 | `r7_disk_document` action `rename` для файла | Успешное переименование по API |
| 9 | Агент не использует pip для файлов на Диске | Ревью диалога / SKILL.md |
