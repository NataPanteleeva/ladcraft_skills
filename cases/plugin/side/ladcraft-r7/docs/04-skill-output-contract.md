# Блок 3: Контракт вывода навыка для кнопок вставки/скачивания

## Назначение

Этот документ задает обязательный контракт ответа агента/навыка для корректной отрисовки действий в `ladcraft-r7`:

- `Вставить в документ` (в начало, в конец, в позицию курсора),
- `Скачать` (`.docx`, `.md`, `.html` fallback).

## UX-контракт (обязательный)

Плагин работает в **intent-gated** режиме:

- кнопки вставки появляются только после явного запроса пользователя (`вставить`, `insert`);
- кнопки скачивания появляются только после явного запроса (`скачать`, `download`, `.md`, `.docx`).

Требование к навыку: после финального отчета добавить подсказку пользователю:

- "Напишите `вставить`, чтобы вставить отчет в документ"
- "Напишите `скачать` для локального .md/.html"
- "Напишите `скачать docx` для Word-файла (или нажмите кнопку «Скачать .docx» в плагине)"

**Не использовать:** «Хотите вставить или скачать docx?» — вопрос не активирует кнопки плагина.

**Не путать с выбором шаблона:** сообщение `sub_roznich.md` — это COMPARE, не download.

Без явного интента пользователя кнопки не показываются.

## Что должен вернуть навык в assistant text

- Финальный user-facing markdown-отчет (summary, таблицы, выводы).
- Без служебных JSON/tool traces в видимой части.
- Рекомендуемые устойчивые маркеры:
  - `Результаты сравнения`
  - `Сравнение завершено`
  - `Расхождений: N`
  - markdown-таблица (`| пункт | шаблон | документ |`).

### CompareReport JSON

`isCompareReport` принимает только объекты с `chatMarkdown` / `chat_markdown` или непустыми `sections[].tables` / `summaryTable`. Поле `schema: doc-compare/v1` **без** контента — недостаточно.

Допустимо и желательно дублировать результат через `r7.task`:

```r7.task
[
  {
    "type": "deliver_inline",
    "data": {
      "fileName": "report.md",
      "mimeType": "text/markdown",
      "content": "# Результаты сравнения\n...",
      "actions": ["download", "paste_text"]
    }
  }
]
```

Поддерживаемые export-типы:

- `deliver_inline`
- `deliver_file`
- `share_link` (опционально)

## Контракт для `.docx`

### Предпочтительно: VFS deliver_file

```r7.task
[
  {
    "type": "deliver_file",
    "data": {
      "fileId": "<vfs_file_id>",
      "fileName": "report.docx",
      "mimeType": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
      "actions": ["download"]
    }
  }
]
```

### Fallback: inline base64 из tool result

Если VFS upload недоступен, допустим tool result с:

- `content_base64`
- `fileName` (`.docx`)
- `mimeType` docx.

## Когда блоки действий не должны появляться

Это нормальное поведение:

- сообщение содержит widget/ожидание ввода;
- сообщение является выбором шаблона;
- нет финального содержательного отчета;
- нет явного интента пользователя (`вставить`/`скачать`).

## Definition of Done для skill-команды

1. Финальный отчет возвращается как markdown в assistant text.
2. После отчета есть инструкция пользователю написать `вставить`/`скачать`.
3. Для `.docx` есть `deliver_file` (или inline base64 fallback).
4. В видимом тексте отсутствуют служебные JSON/tool traces.
5. Smoke-сценарии проходят:
   - отчет без кнопок до интента;
   - `вставить` -> insert-блок;
   - `скачать` после отчета -> download-блок (`.md`/`.html`) на bubble отчета;
   - `скачать docx` после export -> download-блок с `.docx` на bubble export.
